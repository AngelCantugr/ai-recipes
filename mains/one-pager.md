## Purpose
Create compelling one-page documents that concisely communicate ideas, proposals, or project summaries to stakeholders and decision-makers.

## When to Use
- Proposing a new project or initiative
- Seeking approval for technical decisions
- Summarizing project status for executives
- Pitching ideas to leadership
- Creating digestible summaries of complex topics
- Preparing for funding or resource requests

## Template

```
I need to create a one-pager for [topic/project/proposal].

Context:
- Audience: [who will read this - executives, engineers, product, etc.]
- Purpose: [inform/persuade/seek approval/request resources]
- Key decision: [what decision needs to be made, if any]
- Constraints: [timeline, budget, or other relevant constraints]

Topic Overview:
[Brief description of what you want to document]

Key Information:
- Current situation: [problem or opportunity]
- Proposed solution/approach: [your recommendation]
- Benefits: [value proposition]
- Risks/challenges: [what could go wrong]
- Resources needed: [people, time, budget]
- Timeline: [key milestones]
- Success metrics: [how you'll measure success]

Tone: [professional/technical/executive/persuasive]

Please create a one-pager that:
1. Fits on a single page (500-600 words)
2. Leads with the most important information
3. Uses clear headings and structure
4. Emphasizes impact and value
5. Provides just enough detail to make a decision
6. Includes a clear call-to-action or recommendation

Format:
## Title
Brief compelling subtitle

### Problem / Opportunity
[2-3 sentences]

### Proposed Solution
[3-4 sentences with key approach]

### Impact & Benefits
- [Benefit 1]
- [Benefit 2]
- [Benefit 3]

### Approach / Implementation
[3-4 sentences on how it will be done]

### Resources & Timeline
- Team: [size/composition]
- Timeline: [duration]
- Budget: [if applicable]

### Success Metrics
- [Metric 1]
- [Metric 2]

### Risks & Mitigation
- [Risk]: [mitigation]

### Recommendation / Next Steps
[Clear call to action]
```

## Tips
- **Lead with impact**: Start with what matters most to your audience
- **Be specific**: Use numbers, dates, and concrete examples
- **Show, don't just tell**: Include metrics and data points
- **Make it scannable**: Use headers, bullets, bold for key points
- **One page means one page**: Ruthlessly edit to fit 500-600 words
- **Include visuals if helpful**: Simple diagrams or charts add clarity
- **End with action**: Make it clear what you're asking for
- **Tailor to audience**: Technical depth for engineers, business value for executives
- **Highlight risks honestly**: Builds credibility, shows you've thought it through
- **Use active voice**: Makes writing more direct and compelling
- **Test readability**: Can someone skim it in 2 minutes and understand?
- **Version for feedback**: Get input before final version
- **Include contact info**: Make follow-up easy

## Example Use Cases

**New Feature Proposal**:
```
I need a one-pager proposing adding GraphQL API support to our platform.

Context:
- Audience: Engineering VP and Product Director
- Purpose: Get approval for Q3 roadmap inclusion
- Key decision: Approve 6-week project
- Constraints: Team of 3, must not impact current deliverables

Current situation: Partners requesting GraphQL, REST API becoming complex
Proposed: Add GraphQL layer alongside REST
Benefits: Better partner experience, reduced API complexity, modern stack
Risks: Learning curve, maintaining two APIs initially
Resources: 3 engineers, 6 weeks
Timeline: Design 1 week, implement 3 weeks, docs/testing 2 weeks
Success: 3 partners migrate in 3 months, 50% reduction in API calls

Create persuasive one-pager for executive approval.
```

**Technical Architecture Decision**:
```
I need a one-pager documenting our decision to adopt Kubernetes.

Context:
- Audience: Engineering team and infrastructure leadership
- Purpose: Inform and align on architectural direction
- Key decision: Already made, documenting rationale
- Constraints: Migration over 6 months

Document our Kubernetes adoption decision covering:
- Problem: Current deployment complexity, scaling challenges
- Solution: Kubernetes for container orchestration
- Benefits: Better scalability, improved developer experience, industry standard
- Approach: Phased migration starting with new services
- Timeline: 6 month transition
- Training: Team workshops, 2 weeks onboarding per engineer

Tone: Technical but accessible, confident but realistic about challenges.
```

**Incident Summary**:
```
I need a one-pager summarizing yesterday's database outage for leadership.

Context:
- Audience: CTO, VP Eng, Executive team
- Purpose: Inform on what happened, actions taken, prevention plan
- Tone: Factual, accountable, forward-looking

Incident: Database primary failover failed, 45 min outage
Impact: All services down, 10K users affected, ~$15K revenue impact
Cause: Misconfigured replication lag monitoring
Resolution: Manual failover, restored in 45 minutes
Prevention: Automated failover testing, improved monitoring, runbook updates
Timeline: Detection 2min, diagnosis 15min, resolution 28min

Create clear, executive-friendly incident summary.
```
